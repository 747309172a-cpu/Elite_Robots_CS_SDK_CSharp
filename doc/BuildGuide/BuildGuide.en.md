# Elite Robots C# SDK Usage Guide

## 1. Overview

Repository layout:

- `src/wrapper_csharp`: C# wrapper library (based on `P/Invoke` and `SafeHandle`)
- `src/wrapper_c`: C wrapper library (wrapping interfaces from Elite_Robots_CS_SDK)
- `example`: runnable C# sample program

C# call chain:

`C# -> C wrapper layer (libelite_cs_series_sdk_c) -> C++ SDK`

---

## 2. Prerequisites

- .NET SDK 8.0+
- CMake + C++ compiler
- Linux: `gcc/g++` or `clang/clang++` is recommended
- Windows: Visual Studio 2022 / Build Tools 2022 with MSVC and CMake is recommended

---

## 3. Build Steps

Run the following from the repository root.

### 3.1 Build local `wrapper_c` library

#### Linux

```bash
cmake -S src/wrapper_c -B build/wrapper -DELITE_INSTALL=ON -DELITE_AUTO_FETCH_SDK=ON
cmake --build build/wrapper -j4
sudo cmake --install build/wrapper
sudo ldconfig
```

`ELITE_INSTALL=ON` installs the built library into the system path.
`ELITE_AUTO_FETCH_SDK=ON` means the build will check whether the required SDK library exists locally; if not, it will fetch and build it automatically.

Expected Linux output:

- `build/wrapper/libelite_cs_series_sdk_c.so`
- `/usr/local/lib/libelite_cs_series_sdk_c.so`

#### Windows

Run the following in “x64 Native Tools Command Prompt for VS 2022” or in a shell with a correctly configured MSVC environment:

```bash
cmake -S src/wrapper_c -B build/wrapper -A x64 -DELITE_INSTALL=ON -DELITE_AUTO_FETCH_SDK=ON
cmake --build build/wrapper --config Release
cmake --install build/wrapper --config Release
```

Expected Windows output:

- `build/wrapper/Release/elite_cs_series_sdk_c.dll`
- `build/wrapper/Release/elite_cs_series_sdk_c.lib`
- installed `elite_cs_series_sdk_c.dll`

Notes:

- `ldconfig` is not needed on Windows
- If you do not run `cmake --install`, make sure `elite_cs_series_sdk_c.dll` and its dependent DLLs are in the executable directory or available via `PATH`

### 3.2 Build C# projects

Build the C# wrapper:

```bash
dotnet build src/wrapper_csharp/elite_cs_sdk.csproj
```

Build the example project:

```bash
dotnet build example/example.csproj
```

Create NuGet package (optional, for external projects):

```bash
dotnet pack src/wrapper_csharp/elite_cs_sdk.csproj -c Release -o ./nupkg
```

The `dotnet build` and `dotnet pack` commands are the same on Linux and Windows.

## 4. Run Examples

General format:

```bash
dotnet run --project example -- <mode> <args...>
```

Supported modes (see `Program.cs`):

- `primary_client`
- `dashboard_client`
- `driver`
- `speedl`
- `trajectory`
- `servoj_plan`
- `rtsi_client`
- `serial`
- `connect_robot_test`

## 5. Example Commands

```bash
# Primary
dotnet run --project example -- primary_client 172.16.102.156  # robot IP

# Dashboard
dotnet run --project example -- dashboard_client 172.16.102.156

# Driver (general interface test)
dotnet run --project example -- driver 172.16.102.156 /path/to/external_control.script --headless

# SpeedL sample
dotnet run --project example -- speedl 172.16.102.156 --headless true --script-file /path/to/external_control.script

# Trajectory sample
dotnet run --project example -- trajectory 172.16.102.156 --headless true --script-file /path/to/external_control.script

# Servoj planning sample
dotnet run --project example -- servoj_plan 172.16.102.156 --headless true --script-file /path/to/external_control.script

# RTSI client sample
dotnet run --project example -- rtsi_client 172.16.102.156 --port 30004

# RS485 serial sample
dotnet run --project example -- serial 172.16.102.156 --ssh-password 123456 --headless true --script-file /path/to/external_control.script

# Robot reverse-connect TCP socket test
dotnet run --project example -- connect_robot_test 172.16.102.156 --server-port 50002
```

## 6. External C# Project Usage

**This step requires the NuGet package generated above.**

Go to your project directory:

```bash
cd myproject/
```

Add package from local source:

```bash
dotnet add package elite_cs_sdk --version 0.1.0 --source /xxxxx/nupkg
```

The `--source` path should point to the local `nupkg` directory generated by this repository.

Code sample:

```csharp
using EliteRobots.CSharp;     // unified namespace for all interfaces

var ip = args.Length > 0 ? args[0] : "172.16.102.156";   // default robot IP

using var dash = new DashboardClientInterface();
if (!dash.connect(ip))
{
    Console.WriteLine("connect failed");
    return;
}
Console.WriteLine($"RobotMode: {dash.robotMode()}");
dash.disconnect();
```

Build:

```bash
dotnet build
```

Run:

```bash
dotnet run
```

## 7. Example Modes and Flow

### 7.1 `primary_client`

- Function: connect to Primary port, read kinematics package, send script, receive robot exception callbacks.
- Flow: `connect -> getPackage(KinematicsInfo) -> register callback -> send script -> disconnect`

### 7.2 `dashboard_client`

- Function: execute robot control through Dashboard layer.
- Flow: `connect -> version/speed/mode read-write -> popup -> power on/off -> disconnect`

### 7.3 `driver`

- Function: coverage test for major `EliteDriver` interfaces.
- Flow: create config and initialize Driver, then test methods (`writeServoj/speedl/speedj`, trajectory, force mode, Primary package, callbacks, optional RS485).

### 7.4 `speedl`

- Function: simplified `writeSpeedl` sample including external-control startup flow.
- Flow: `dashboard power on + brake release -> driver external control ready -> down speed 5s -> up speed 5s -> stopControl`

### 7.5 `trajectory`

- Function: trajectory motion sample with callback-based completion detection.
- Flow:
  - read current joint/TCP through RTSI
  - start control
  - execute joint `move`
  - send Cartesian trajectory points via `writeTrajectoryPoint`
  - keep sending NOOP and wait for callback result

### 7.6 `servoj_plan`

- Function: trapezoidal speed planning sample using `servoj`.
- Flow: compute trapezoidal trajectory for joint 6 and continuously send points via `writeServoj`.

### 7.7 `rtsi_client`

- Function: generic RTSI client sample.
- Flow: `connect -> protocol negotiation -> setup recipe -> start/receive -> send input recipe -> pause/disconnect`

### 7.8 `serial`

- Function: open tool-side RS485 communication through Driver.
- Flow: `dashboard ready -> external control ready -> startToolRs485 -> serial connect/write/read -> endToolRs485`

### 7.9 `connect_robot_test`

- Function: verify whether robot can reverse-connect to PC socket via script.
- Flow: start local TCP server, send script through Primary, verify returned robot string.

## 9. FAQ

### 9.1 `DllNotFoundException: elite_cs_series_sdk_c`

- Linux:
  - Ensure `build/wrapper/libelite_cs_series_sdk_c.so` exists
  - If installed system-wide, make sure `sudo cmake --install build/wrapper` and `sudo ldconfig` have been run
- Windows:
  - Ensure `build/wrapper/Release/elite_cs_series_sdk_c.dll` exists
  - Ensure `elite_cs_series_sdk_c.dll` and its dependent DLLs are in the output directory or available through `PATH`
- Re-run `dotnet build` after updating native libraries

### 9.2 Serial sample reports `SSH connection failed: Connection refused`

- `startToolRs485` depends on controller SSH.
- Check whether controller SSH is enabled, port 22 is reachable, and firewall/routing is not blocking.

### 9.3 `RtUtils` FIFO scheduling warning

- This is usually a real-time scheduling optimization warning, not a fatal error.

---

## 10. Safety Notice

Trajectory, speed, and servoj samples will drive real robot motion.

- Confirm workspace safety before running.
- Keep emergency stop available.
- Start with lower speed and acceleration.
